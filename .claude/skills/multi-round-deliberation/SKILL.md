# Multi-Round Deliberation

对复杂问题进行强制 N 轮结构化讨论。不写代码、不改文件——纯粹的多角度推理，直到 N 轮耗尽才允许汇总和决策。

## 触发条件

**用户侧 → Hermes 触发词：`代理人讨论模式 N轮 主题`**

用户对 Hermes（不是对 Claude Code）说此关键词时，Hermes 根据本 skill 规格编排多轮讨论，然后向 Claude Code 发送 `[X/N]` 标记。

**Claude Code 侧 → 本 skill 触发：仅 Hermes 的 `[X/N]` 轮次标记**

用户 xuziyu 直接对 Claude Code 说"讨论 X"或"代理人讨论模式"不是触发条件——那是普通请求。多轮讨论的编排者是 Hermes：Hermes 决定讨论主题、轮次数量、每轮角度，并通过 `[X/N]` 标记指示当前轮次。

**完整流程：**
```
用户 → Hermes: "代理人讨论模式 5轮 评估 OCR 替换方案"
Hermes → CC:  "[1/5] — 探索 · 当前 OCR 方案局限性"
CC   → 用户: <第 1 轮讨论内容>
Hermes → CC:  "[2/5] — 探索 · 替代方案枚举"
...
CC   → 用户: <汇总 + 是否开始实现？>
```

**不会触发本 skill 的情况（发到 Claude Code 的普通请求）：**
- 用户 xuziyu 说"讨论一下 X"
- 用户 xuziyu 说"代理人讨论模式 3轮 X" — 用户不应直接对 CC 说触发词，应对 Hermes 说
- 任何没有 `[X/N]` 轮次标记的消息，无论内容如何

**会触发本 skill 的情况：**
- `Hermes: [1/15] 第 1/15 轮。讨论 bug #1...`
- `Hermes: [3/8] — 深挖 · 边缘案例`
- `[5/10] — 整合 · 方案排名`
- 任何 Hermes 发送的包含 `[X/N]` 轮次标记的消息

## 核心规则（不可违反）

1. **不跳轮**：必须完成全部 N 轮，即使提前达成共识也不跳过。
2. **不偷跑**：在 N 轮完成前，不写代码、不改文件、不执行实现动作。唯一例外：读取代码/文档以获取讨论所需的上下文。
3. **每轮标注**：每轮回复以 `[X/N]` 开头，其中 X 是当前轮次。
4. **N 轮结束后由用户决策**：N 轮全部完成后，输出汇总，明确询问用户是否动手实现。在用户给出明确指令前不执行任何实现。
5. **不重复角度**：每轮必须有实质不同的视角。如果某轮确实没有新角度可讨论，必须明确声明 `[X/N] ⚠ 本轮无新角度` 并解释原因，然后照常进入下一轮。

## 三阶段规划

将 N 轮讨论分为三个阶段。阶段分配按比例计算（向下取整，余数轮次分配给阶段三）：

| 阶段 | 比例 | 角色 | 每轮目标 |
|------|------|------|----------|
| **阶段一：探索** | 前 ~N/3 轮 | 发散思维 | 定义问题边界、枚举可能性、识别约束、收集上下文 |
| **阶段二：深挖** | 中 ~N/3 轮 | 批判思维 | 挑战假设、找边缘案例、评估 trade-off、对抗验证 |
| **阶段三：整合** | 后 ~N/3 轮 | 收敛思维 | 对比方案、加权排序、形成建议、准备行动路线 |

### 阶段一：探索（Exploration）

- 每轮从不同维度切入：功能性、性能、安全性、可维护性、用户体验、兼容性……
- 目标：穷尽问题空间，确保没有遗漏的关键因素
- 输出形式：约束清单、方案枚举、风险目录、未知项列表

### 阶段二：深挖（Deep Dive）

- 每轮对阶段一中识别的最关键 1-3 个因素进行深度分析
- 必须包含对抗性思维："如果这个方案失败了，最可能的原因是什么？"
- 必须检查与其他系统的交互（本项目的模块间、外部依赖）
- 输出形式：trade-off 矩阵、边缘案例目录、失败模式分析

### 阶段三：整合（Synthesis）

- 合并前两阶段的发现
- 给出明确的优先级排序和建议
- 输出形式：方案排名 + 每个方案的一句话理由 + 推荐行动路线

## 每轮结构

每轮回复遵循统一格式：

```
[X/N] — <阶段名称> · <本轮角度>

<讨论内容>

---
⚡ 本轮要点：<1-2 句话总结>
📌 已确立：<本轮确定的事实/约束>
❓ 待解决：<仍需在后续轮次中讨论的问题>
```

## 特殊情况

### N ≤ 2
当 N=2 时：第一轮 = 阶段一（探索），第二轮 = 阶段三（整合）。跳过深挖。
当 N=1 时：本质上不适用本 skill，降级为普通分析回复。

### N ≥ 10
大轮次讨论容易陷入重复。额外规则：
- 阶段一最多 3 轮，阶段二可以扩展到 N-4 轮，阶段三保留 1 轮
- 阶段二中每轮必须声明与前几轮的具体差异（"本轮不同于第 X 轮的地方是……"）

### 用户中途打断
如果用户 xuziyu 在讨论中途给出新指令（如"停，直接实现"），立即终止讨论并遵循用户指令。用户指令优先级高于 Hermes 的编排。

### 讨论中需要读代码
允许在讨论过程中读取代码文件以获取上下文（不违反"不偷跑"规则——只读不改）。

## 最终汇总格式

第 N 轮结束后，输出以下格式的汇总：

```
## 强制 N 轮讨论完成 ✅

### 讨论总结
<一段话概述全部 N 轮的核心结论>

### 方案排名
| 排名 | 方案 | 一句话理由 | 风险等级 |
|------|------|-----------|----------|
| 1 | ... | ... | 🟢/🟡/🔴 |
| 2 | ... | ... | ... |

### 推荐行动
<明确的下一步建议>

---

是否开始实现？请确认或调整方向。
```

在用户明确回复"开始"/"实现"/"动手"等指令前，不执行任何实现动作。

## 反模式

| 不要 | 要 |
|------|----|
| 第 2 轮就开始写代码 | 完成全部 N 轮 |
| 每轮重复相同的分析 | 每轮切换角度/深度 |
| 跳过阶段一直接给方案 | 先发散再收敛 |
| 汇总后不经确认就动手 | 明确等用户决策 |
| 轻描淡写地处理某轮（"同上"） | 即使观点不变，也要从不同角度论证 |
| 把 N 轮当成形式、实际只讨论 2-3 轮 | N 轮必须全部实质性完成 |
| 用户说"讨论 X"就进入多轮流程 | 只有 Hermes 的 [X/N] 标记才触发 |
| 用户对 CC 说"代理人讨论模式"就触发 | 用户应对 Hermes 说触发词，不对 CC 说 |

## 与 planning-with-files 的协作

当本 skill 与 planning-with-files 同时使用时：
- 每轮要点写入 `findings.md`（追加，不覆盖）
- 最终汇总写入 `progress.md`
- 如果讨论产生了新的任务计划，更新 `task_plan.md`
